内存问题的疑难

一旦确定内存有问题，应该再尚未开始清理内存的地方寻找解决问题的机会。可以通过两种方式做：

  • 使用delete关键字从内存中移除不再需要的js对象
  • 从网页的dom树上移除不再是必需的节点。


拆分初始化加载

1. 通过拆分来节省下载量

2. 未定义标识符和竞争状态

例如对下拉菜单的js延后加载，建立一个下载js和用户点击菜单之间的竞争状态。大多数情况下，js会赢得这个竞争状态。

在延迟加载的代码与用户界面元素相关联的情况下：

  • 可通过改变元素展现。如提示“加载中”来提醒用户功能未加载完成。
  • 或者在延迟加载的代码里绑定界面元素的时间处理程序。本栗中菜单会被初始化成静态文本，延迟加载的代码包括菜单的功能和时间的绑定。在ie中用attachEvent实现，其他浏览器用addEventListener。
    在延迟加载的代码不与界面元素相关联的情况下，可以使用stub(桩)函数解决。该函数是一个与原函数名称相同但是函数体空的函数，或是用一些临时代码顶替原有内容的函数。加载完成时原函数再覆盖住stub函数。更简单的方法是给每一个被引用但又被降级为延迟下载的函数创建一个桩函数。如有需要桩函数可以返回一个桩值，例如一个空字符串。如用户在完整的函数下载完成前尝试使用这个功能，将没有反应。稍高级的解决方案是，用桩函数记录用户的请求，并在js完成加载时调用相应的代码。


无阻塞加载脚本

  • 脚本并行加载

  • 外部脚本技术

1. XHR Eval

从服务端获取脚本，响应完成时通过eval命令执行内容。缺陷是获取的脚本必须部署在和主页面相同的域中。

2. XHR注入

通过创建一个script的DOM元素，然后把XMR的响应注入script中执行js，在某些情况下使用eval比这种机制慢。缺陷是获取的脚本必须部署在和主页面相同的域中。

3. Script in iframe

该技术利用iframe无阻塞加载js。如：

Script in iframe
    http://stevesouders.com/cuzillion/?ex=10012&title=Script+in+iframe
实现过程完全在HTML中完成
    <iframe src'A.html' width=0 height=0 frameborder=0 id=frame1></iframe>

因为iframe返回的是html文档，所以要在html文档中把外部脚本转换成行内脚本，需要修改js来创建他们之间的关联。两种方法：

1. 通过frames数组活document.getElementById来获取引用iframe的js标识符。
//使用frames来访问主页上的iframe
window.frames[0].createNesDiv();
//使用getElmentById访问主页上的iframe
document.getEementById('frame1').contentWindow.createNewDiv();
2. iframe使用parent变量引用父页面：
//在iframe中使用"parent"访问主页面
function createNewDiv() {
    var NewDiv = parent.document.creatElement('div');
    parent.document.body.appendChild(newDiv);
}

缺陷是要求iframe URL和主页面同域。Iframe存在自身的消耗，是开销最高的dom元素，至少比普通dom元素高出一个数量级。

4. Script DOM Element

动态创建script DOM元素并设置其src属性。该技术允许跨域获取脚本，且无需像XHR Eval或Script in iframe需要重构。

var se = document.createElement('sript');
se.src = 'js地址';
document.getElementByTagName('head')[0].appendChild(se);

5. Script Defer

defer属性是非常简单的防止脚本阻塞行为的方法，但只在ie，saf4，chr2，ff3.5中是并行加载脚本。跨域。

<script defer src="A.js"></script>

6. document.write Script Tag

该技术只在ie，saf4，chr2，op中是并行加载脚本。跨域。（不推荐）

确保（避免）按顺序进行

Script DOM Element技术能保证脚本按列出的顺序执行。
XHR注入技术和XHR Eval技术加载顺序无所谓。

最佳方案

可通过函数function loadScript(url, bPreserveOrder, bShowBusy);来实现，确保按最佳方案加载脚本。

  • 不同域、无序
    例如广告和空间的脚本
  • 不同域、顺序
    例如从不同的服务器中下载多个存在依赖关系的js文件的页面
  • 同域、无序、无忙指示器
    推荐XHR注入，应用于希望在后台下载js文件的网页。
  • 同域、无序、有忙指示器
    推荐XHR注入，需要额外的js来激活忙指示器，当XHR发出激活状态栏和光标，XHR返回时恢复。
  • 同域、顺序、无忙指示器
    推荐XHR注入，适用于有多个存在内部依赖的脚本在后台下载的页面。
  • 同域、顺序、有忙指示器
    ie的Script和ff的Script DOM Element技术是首选方案。


整合异步脚本

整合技术

整合行内代码和依赖行内代码的外部脚本。

1. 硬编码回调

在外部脚本底部添加一个对init的调用。该方法改变回调接口时需要调整外部脚本，不灵活。

2. window Onload

该技术需要确保外部脚本在window.onload之前下载执行就能保持执行顺序。有两个缺点：

  • 首先必须确认异步脚本是通过阻塞onload事件的方式加载的。（首选Script in Iframe）
  • 其次，可能会导致行内代码延迟执行。

3. 定时器

可以使用setTimeout方法。
该方法会稍增加维护成本，因为需要通过外部脚本的特定标识符来判断脚本是否已经加载完毕，如果外部脚本的标识符变了，行内代码也需要同步更新。

4. script Onload

该技术是整合异步加载外部脚本和行内脚本的首选。

5. Degrading Script Tags

该方法更干净、清晰、安全，但需要修改外部脚本，在使用第三方js库的时候很难做到。

异步加载多个外部脚本

1. Managed XHR

该技术解决了跨主流浏览器的兼容性问题，但是由于XMLHttpRequest的同源规则，如果外部脚本部署在与主页面不同的域下，该技术就无法工作。

2. DOM Element 和 Doc Write

结合Script DOM Element和document.write ScriptTag技术完成主要目标：外部脚本的执行顺序在所有浏览器下都能保持，行内代码和它依赖的外部脚本成功整合。但异步加载在各个浏览器中实现的程度有所有不同。

没有简单的跨浏览器方案可以异步加载多个脚本，还同时保持执行顺序。

综合方案

1. 单个脚本

整合异步加载单个脚本最佳技术Script Dom Element和用于整合行内代码和单个外部脚本的最好选择Scipt Onload。

2. 多个脚本

EFWS.Script.loadScripts函数。


布置行内脚本

避免行内脚本阻塞下载和渲染，有以下几种方式：

1. 把行内脚本移动到底部

这种方式虽然会避免阻塞下载，但是依旧不能避免阻塞渲染的问题。故，只能解决执行脚本时间少于300毫秒的问题。

2. 异步启动执行脚本

简延迟值为0毫秒时用setTimeout，ie可以逐步渲染（最好的方法）。脚本时间过长，更好的选择是onload。
最佳方案是每300毫秒左右挂起或者使用setTimeout，但这种方法需要大规模的重新设计和重构代码。

3.使用script的defer属性

只有ie和ff3.1+支持，允许浏览器继续解析和渲染页面的同时延迟执行行内脚本。defer属性允许两张图片并行下载，页面总加载时间为5s，5
s后脚本执行完成时页面才开始渲染。（基准页面为6s）

样式表阻塞行内脚本

由于css和js的解析顺序。样式表阻塞行内脚本，因为行内脚本可能含有依赖于样式表中样式的代码。
这也就导致了在样式表后面的行内脚本会阻塞所有后续资源的下载
所以，调整行内脚本的位置，使其不出现在样式表和任何其他资源之间。具体表现为，：

  • 把行内脚本放在样式表之前

    如果其他资源是脚本，行内脚本和外部脚本之间可能会有代码依赖。

  • 把行内脚本移到可见资源之后

    如果确认没有代码依赖。


编写高效的js

作用域

执行js代码时，js引擎会创建一个执行上下文，有时也被称为作用域。执行上下文设定了代码执行时所处的环境。js引擎会在页面加载后创建一个全局的执行上下文，然后每执行一个函数时都会创建一个对应的作用域，最终建立一个作用域的堆栈，当前起作用的作用域在堆栈的最顶部。

每个执行上下文都有一个与之关联的作用域链，用于解析标识符。作用域链包含一个或多个变量对象，这些对象定义了执行上下文作用域内的标识符。全局执行上下文的作用域链中只有一个变量对象，它定义了js中所有可用的全局变量和函数。当函数被创建（不是执行），js引擎会把创建时执行上下文的作用域链赋给函数的内部属性（[scope]），当函数被执行时，js引擎会创建一个活动对象，并在初始化时给this,arguments,命名参数和该函数的所有局部变量赋值。活动对象会出现在执行上下文作用域链顶端，紧接其后的是函数属性中的对象。

1. 使用局部变量

局部变量是js中读写最快的标识符。
标识符在作用域链中的位置越靠上，存取的速度就越快。
全局变量对象始终是作用域链中最后一个对象，对其的解析也总是最耗时的。

2. 增长作用域链

需避免因不必要地增长作用域链而无意中导致执行速度变缓慢。

  • with语句
    用于将对象属性作为局部变量来显示，使其便于访问。但在作用域链中增加的额外对象会影响对局部标识符的解析，建议避免使用。
  • try-catch语句中的catch从句
    行为方式类似with语句，也是在作用域链顶部增加一个对象，该对象包含了有catch指定命名的异常对象。

数据存取

脚本中4种存取数据的地方：

1. 字面量值
2. 变量
3. 数组元素
4. 对象属性

将函数中使用超过一次的对象属性或数组元素存储为局部变量。（查找属性比存取局部变量更耗时）
随着数据结构深度的增加，它对数据存取速度的影响也跟着变大。

流控制

1. 快速条件

  • if语句
  1. 1-2个离散值
  2. 大量的值能容易地分到不同的区间范围
  • switch语句
  1. 超过两个少于10个离散值
  2. 条件值是非线性的，无法分离出区间范围。
  • 数据查询
  1. 超过10个值需要判断
  2. 条件对应的结果是单一的，而不是一系列操作。

2. 快速循环

  • 将循环变量递减到0
    注意，一旦循环变量等于0，结束条件就会变为假
    小心使用数组原生的inDexOf方法，这个方法遍历数组成员的耗时可能会比使用普通的循环还长。
  • 避免for-in循环
    它的的结束条件无法改变，而且遍历属性的顺序也无法改变，需要从一个特定的对象中解析每个枚举的属性，故而更加耗时。
    只有在已知需要遍历的对象属性集时才使用这种方法。
    处理诸如JSON对象这样的未知属性集时，依然必须使用for-in循环。
  • Duff策略
    主要用于处理大数组，仅当性能瓶颈是由循环处理大量元素项所引起时才考虑。

字符串

1. 连接字符串

用户主要用ie6-7时，使用数组技术连接字符串才值得。
ie处理字符串链接的速度比其他浏览器慢很多，可以先将其存储到数组中，然后调用join()方法合并他们

2. 裁剪字符串

裁剪字符串的速度只有在整个执行过程中裁剪的频率足够大才重要。下面的栗子在处理长字符串时就明显更快。

//去除空白/^\s+|\s+$/g/
function trim(text) {
    text = text.replace(^\s+/,'');//删除头部空白
    for(var i = text.length - 1;i >= 0; i--) {//清楚尾部空白后退出循环
        if (/\S/.text(text.charAt(i))) {
        text = text.substring(0, i + 1);
        }
    }
    return text;
}

避免运行时间过长的脚本

浏览器会限制js可以运行的最长时间，以下原因可能会影响js执行总时间

  1. 过多的DOM交互
  2. 过多的循环
  3. 过多的递归
解决方法：
  1. 定时器setTimeout
  2. 用于挂起的定时器模式
  • 利用定时器拆分数组的函数chunk()
  • 利用定时器执行大型运算中小而有序的部分。
