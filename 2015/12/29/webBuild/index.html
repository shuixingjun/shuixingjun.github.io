内存问题的疑难

一旦确定内存有问题，应该再尚未开始清理内存的地方寻找解决问题的机会。可以通过两种方式做：

  • 使用delete关键字从内存中移除不再需要的js对象
  • 从网页的dom树上移除不再是必需的节点。

    拆分初始化加载

    1. 通过拆分来节省下载量

    2. 未定义标识符和竞争状态

    例如对下拉菜单的js延后加载，建立一个下载js和用户点击菜单之间的竞争状态。大多数情况下，js会赢得这个竞争状态。
    在延迟加载的代码与用户界面元素相关联的情况下：
  • 可通过改变元素展现。如提示“加载中”来提醒用户功能未加载完成。
  • 或者在延迟加载的代码里绑定界面元素的时间处理程序。本栗中菜单会被初始化成静态文本，延迟加载的代码包括菜单的功能和时间的绑定。在ie中用attachEvent实现，其他浏览器用addEventListener。
    在延迟加载的代码不与界面元素相关联的情况下，可以使用stub(桩)函数解决。该函数是一个与原函数名称相同但是函数体空的函数，或是用一些临时代码顶替原有内容的函数。加载完成时原函数再覆盖住stub函数。更简单的方法是给每一个被引用但又被降级为延迟下载的函数创建一个桩函数。如有需要桩函数可以返回一个桩值，例如一个空字符串。如用户在完整的函数下载完成前尝试使用这个功能，将没有反应。稍高级的解决方案是，用桩函数记录用户的请求，并在js完成加载时调用相应的代码。

    无阻塞加载脚本

  • 脚本并行加载

  • 外部脚本技术

    1. XHR Eval

    从服务端获取脚本，响应完成时通过eval命令执行内容。缺陷是获取的脚本必须部署在和主页面相同的域中。

    2. XHR注入

    通过创建一个script的DOM元素，然后把XMR的响应注入script中执行js，在某些情况下使用eval比这种机制慢。缺陷是获取的脚本必须部署在和主页面相同的域中。

    3. Script in iframe

    该技术利用iframe无阻塞加载js。如：

    Script in iframe
    http://stevesouders.com/cuzillion/?ex=10012&title=Script+in+iframe
实现过程完全在HTML中完成
    <iframe src'A.html' width=0 height=0 frameborder=0 id=frame1></iframe>

因为iframe返回的是html文档，所以要在html文档中把外部脚本转换成行内脚本，需要修改js来创建他们之间的关联。两种方法：

1. 通过frames数组活document.getElementById来获取引用iframe的js标识符。
//使用frames来访问主页上的iframe
window.frames[0].createNesDiv();
//使用getElmentById访问主页上的iframe
document.getEementById('frame1').contentWindow.createNewDiv();
2. iframe使用parent变量引用父页面：
//在iframe中使用"parent"访问主页面
function createNewDiv() {
    var NewDiv = parent.document.creatElement('div');
    parent.document.body.appendChild(newDiv);
}

缺陷是要求iframe URL和主页面同域。Iframe存在自身的消耗，是开销最高的dom元素，至少比普通dom元素高出一个数量级。

4. Script DOM Element

动态创建script DOM元素并设置其src属性。该技术允许跨域获取脚本，且无需像XHR Eval或Script in iframe需要重构。

var se = document.createElement('sript');
se.src = 'js地址';
document.getElementByTagName('head')[0].appendChild(se);

5. Script Defer

defer属性是非常简单的防止脚本阻塞行为的方法，但只在ie，saf4，chr2，ff3.5中是并行加载脚本。跨域。

<script defer src="A.js"></script>

6. document.write Script Tag

该技术只在ie，saf4，chr2，op中是并行加载脚本。跨域。（不推荐）

确保（避免）按顺序进行

Script DOM Element技术能保证脚本按列出的顺序执行。
XHR注入技术和XHR Eval技术加载顺序无所谓。

最佳方案

可通过函数function loadScript(url, bPreserveOrder, bShowBusy);来实现，确保按最佳方案加载脚本。

  • 不同域、无序
    例如广告和空间的脚本
  • 不同域、顺序
    例如从不同的服务器中下载多个存在依赖关系的js文件的页面
  • 同域、无序、无忙指示器
    推荐XHR注入，应用于希望在后台下载js文件的网页。
  • 同域、无序、有忙指示器
    推荐XHR注入，需要额外的js来激活忙指示器，当XHR发出激活状态栏和光标，XHR返回时恢复。
  • 同域、顺序、无忙指示器
    推荐XHR注入，适用于有多个存在内部依赖的脚本在后台下载的页面。
  • 同域、顺序、有忙指示器
    ie的Script和ff的Script DOM Element技术是首选方案。
